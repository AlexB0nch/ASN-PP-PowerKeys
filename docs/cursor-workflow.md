# AI-workflow в Cursor: архитектор + субагенты без потери контекста

Это руководство описывает, как вести разработку **PPT PowerKeys** в Cursor так, чтобы:

- субагенты **писали код** (в т.ч. параллельно);
- отдельная **архитекторская сессия** планировала и проверяла PR;
- **контекст не терялся** между сессиями и агентами.

> Важно про версии Cursor: в Cursor 2.1 удалены **Custom Modes** и **Memories**. Поэтому «режимы» мы делаем через **субагентов**, а «память» — через **файлы в репозитории** (правила, ADR, планы). Это надёжнее «вечного чата».

---

## 1. Ключевая идея: контекст живёт в git, а не в чате

LLM не помнят ничего между запусками, а окно контекста ограничено. Поэтому durable-знание мы выносим в файлы, которые Cursor подхватывает автоматически:

| Артефакт | Файл(ы) | Когда применяется |
|----------|---------|-------------------|
| Общие инструкции агентам | `AGENTS.md` | Всегда (читается автоматически) |
| Правила проекта | `.cursor/rules/*.mdc` | По `alwaysApply` / `globs` / по запросу |
| Архитектура | `docs/architecture.md` | По ссылке/`@`-mention |
| Решения и их причины | `docs/adr/*` | По ссылке/`@`-mention |
| Планы фич | сохранённые планы Plan Mode (`Save to workspace`) | Передаются builder'у |
| Правила ревью | `.github/review/CHECKLIST.md`, `.cursor/BUGBOT.md` | При review / в Bugbot |

Благодаря этому **любая новая сессия любого агента стартует с одним и тем же контекстом** — ничего не теряется, даже если закрыть чат.

Уже настроено в репозитории:

```
AGENTS.md                      # durable-инструкции для всех агентов
.cursor/
├── rules/
│   ├── 00-project-overview.mdc   # alwaysApply: обзор проекта
│   ├── 10-architecture.mdc       # globs: src/** — принципы архитектуры
│   ├── 20-csharp-vsto.mdc        # globs: *.cs — конвенции C#/Interop
│   └── 30-git-workflow.mdc       # alwaysApply: git-процесс
├── agents/
│   ├── architect.md              # read-only: планирование
│   ├── builder.md                # пишет код
│   └── reviewer.md               # read-only: ревью
├── commands/
│   ├── plan-feature.md           # /plan-feature
│   ├── build-from-plan.md        # /build-from-plan
│   └── review-pr.md              # /review-pr
└── BUGBOT.md                     # правила ревью PR на стороне GitHub
```

## 2. Роли = субагенты (а не «режимы»)

Субагент в Cursor — это markdown-файл с YAML-фронтматтером в `.cursor/agents/`. У каждого свой системный промпт, своя модель и флаг `readonly`. Это и есть наша замена удалённым Custom Modes.

| Роль | Файл | `readonly` | Что делает |
|------|------|-----------|------------|
| **Architect** | `.cursor/agents/architect.md` | `true` | Проектирует, декомпозирует задачи, готовит планы. Код не пишет. |
| **Builder** | `.cursor/agents/builder.md` | `false` | Пишет код по плану, коммитит, открывает PR. |
| **Reviewer** | `.cursor/agents/reviewer.md` | `true` | Проверяет PR по чеклисту. Код не меняет. |

Вызов субагента:
- явно: `@architect ...`, `@builder ...`, `@reviewer ...` или `/architect ...`;
- или главный агент делегирует сам, ориентируясь на поле `description`.

`readonly: true` у архитектора и ревьюера гарантирует, что они **не редактируют файлы** — то есть архитекторская сессия физически не может «случайно» писать код. Именно это даёт безопасное разделение «планирую/проверяю» vs «пишу».

## 3. Как держать архитектора в отдельной сессии и не терять контекст

Рекомендуемая схема — **Agents Window** (открывается `Cmd/Ctrl+Shift+P → Open Agents Window`): в нём удобно держать несколько параллельных агентов.

```
┌─ Agents Window ───────────────────────────────────────────┐
│                                                            │
│  [Сессия A: Architect]      [Сессия B: Builder #1]         │
│  read-only, Plan Mode       пишет код (worktree/ветка)     │
│  планирует S01-004          реализует S01-002              │
│                                                            │
│  [Сессия C: Builder #2]     [Сессия D: Reviewer]           │
│  реализует S01-005          ревьюит PR #N                  │
└────────────────────────────────────────────────────────────┘
        ▲                              ▲
        └── все читают один и тот же durable-контекст ──┘
            (AGENTS.md, .cursor/rules, docs/adr, планы)
```

Что обеспечивает «непрерывность контекста архитектора»:

1. **Externalize context.** Архитектор фиксирует решения не «в голове чата», а в файлах: сохранённый план (`Save to workspace`) + при необходимости новый ADR в `docs/adr/`. Эти файлы переживут закрытие сессии.
2. **Worktrees для изоляции.** Каждый builder работает в своём git worktree/ветке, поэтому правки разных задач не смешиваются и не «засоряют» сессию архитектора.
3. **`@Past Chats`.** Если нужно дотянуть конкретное прошлое обсуждение в новую сессию — подключите его через `@Past Chats` (ближайший аналог «resume» для чатов).
4. **Resume субагента.** Каждое выполнение субагента возвращает agent ID — его можно возобновить позже с сохранённым контекстом.

> Не пытайтесь «держать один вечный чат живым» — это упирается в окно контекста. Надёжнее: durable-файлы + при необходимости `@Past Chats`/resume.

## 4. Конвейер: план → код → ревью → PR

```
 ┌──────────┐   план в      ┌──────────┐   ветка+PR    ┌──────────┐   ревью     ┌────────┐
 │ Architect│──workspace──▶ │ Builder  │────────────▶  │ Reviewer │──────────▶  │  Merge │
 │ (Plan)   │               │ (код)    │               │ + Bugbot │             │  main  │
 └──────────┘               └──────────┘               └──────────┘             └────────┘
   readonly                   пишет код                  readonly
```

### Шаг 1 — Планирование (Architect, Plan Mode)
1. В отдельной сессии вызовите `/plan-feature` или `@architect` и укажите задачу (ID из `sprints/sprint-NN-*/backlog.md`).
2. Архитектор в **Plan Mode** исследует код, задаёт уточняющие вопросы и готовит пошаговый план + acceptance criteria + риски.
3. Нажмите **Save to workspace** — план становится durable-артефактом для исполнителя.
4. Значимое решение → новый ADR в `docs/adr/`.

### Шаг 2 — Реализация (Builder)
1. В **отдельной** сессии вызовите `/build-from-plan` или `@builder` и передайте сохранённый план.
2. Builder создаёт ветку `cursor/<кратко>-7151`, пишет код по `.cursor/rules/`, коммитит по Conventional Commits.
3. Для параллельной работы запускайте несколько builder-сессий в Agents Window (каждая — своя задача/worktree).
4. Builder открывает PR в `main` по шаблону, указывает Sprint и Task ID.

### Шаг 3 — Ревью (Reviewer + Bugbot)
1. **Локально:** `/review-pr` или `@reviewer` — проверка по `.github/review/CHECKLIST.md`, вердикт Approve/Request changes.
2. **На GitHub:** Bugbot автоматически ревьюит PR по `.cursor/BUGBOT.md` (или вручную — комментарий `bugbot run` / `cursor review`).
3. Чинить найденное можно ссылками **Fix in Cursor**, либо включить Bugbot **Autofix** (он поднимет cloud-агента и запушит фикс).

### Шаг 4 — Merge
Squash merge в `main` после approve и зелёного ревью; задача → `Done`.

## 5. Локально vs Cloud (важно для VSTO)

| Где | Для чего годится | Ограничение |
|-----|------------------|-------------|
| **Локальные агенты** (ваш Windows) | Сборка VSTO, отладка в PowerPoint, финальная проверка | Нужен Windows + Office |
| **Субагенты** (в вашей сессии) | План, код, ревью, рефактор | Не запускают Office |
| **Cloud-агенты** (`/in-cloud`, `@cursor` на PR) | Правки кода, рефактор, docs, babysitting PR | VM на Linux — **не соберут/не протестируют** против Office |

Поэтому: **проектирование, написание кода и ревью** делаем через субагентов/cloud, а **сборку и тестирование в PowerPoint** — всегда локально на Windows (см. `setup/environment/`). Builder не должен утверждать, что «протестировал в Office», если это не так — он помечает, что требуется ручная проверка.

## 6. Первичная настройка (один раз)

1. **Правила и роли** — уже в репозитории (`.cursor/`, `AGENTS.md`). Cursor подхватит их автоматически после `git pull`.
2. **Bugbot** — включить репозиторий в [Bugbot dashboard](https://cursor.com/dashboard/bugbot) (нужны права; ревью PR на стороне GitHub).
3. **Agent Review** (опционально) — локальное ревью до PR: Cursor Settings → Agents → Agent Review.
4. **Source control для cloud-агентов** (опционально) — админ подключает GitHub в настройках Cursor, чтобы работали `/in-cloud` и `@cursor`.
5. **Индексация docs** (опционально) — добавьте справку по Office/VSTO через `@Docs → Add new doc`, чтобы агенты ссылались на актуальный API.

## 7. Чеклист «контекст не теряется»

- [ ] Новое значимое решение зафиксировано как **ADR** (`docs/adr/`).
- [ ] План фичи сохранён в workspace (**Save to workspace**), а не только в чате.
- [ ] Изменения правил/конвенций отражены в `.cursor/rules/*.mdc` или `AGENTS.md`.
- [ ] Статусы задач обновлены в `sprints/.../backlog.md`.
- [ ] PR ссылается на Sprint и Task ID.

Если всё это соблюдается — любую сессию можно закрыть и открыть заново, а архитектор/builder/reviewer стартуют с полным контекстом.
